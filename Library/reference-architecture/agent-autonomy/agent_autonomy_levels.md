# Agent Autonomy Levels: A0 to A4

## Purpose

This document defines a taxonomy of agent autonomy levels for AI systems deployed in financial institutions. Autonomy level is treated as a design decision, not a consequence of how capable the underlying model is. The same model can operate at different autonomy levels depending on how the surrounding system is built and how human oversight is enforced.

---

## Autonomy Level Definitions

| Level | Name | Description | Key Characteristics | Security Focus | FSI Examples |
|:---|:---|:---|:---|:---|:---|
| **A0** | No Agency | The model answers questions and produces output only. It cannot take any action, access any system, or do anything on behalf of the user. A human reads the output and decides what to do with it. | - No ability to access tools, APIs, or external systems<br>- No memory between conversations<br>- No system state mutation of any kind<br>- Human carries out all actions based on the output<br>- Works as a question and answer or text generation tool only | - Ensure inputs cannot be crafted to manipulate or extract sensitive information from the model<br>- Ensure outputs cannot be used to leak confidential data<br>- Prevent personal or sensitive information from being included in prompts<br>- Control who can access and use the model<br>- Log and retain conversations in line with data policies | - Analyst using an LLM to summarise earnings call transcripts<br>- Compliance officer querying an LLM for regulatory interpretation<br>- Credit analyst using a model to draft initial commentary on a borrower profile |
| **A1** | Tool-Calling Agent | The model can look things up or retrieve information by calling specific, approved tools. However, it cannot do anything without a human explicitly approving each action first. One action at a time, one approval at a time. | - Can only use tools that have been explicitly approved and registered<br>- Every action must be approved by a human before it is carried out<br>- Cannot chain multiple actions together without human approval at each step<br>- No memory between sessions<br>- Cannot hand off tasks to other agents | - Only approved tools can be called by the agent<br>- Every tool call requires a human to confirm before it executes<br>- Tools are given the minimum level of access needed to do their job<br>- Detect and block attempts to manipulate the agent into calling unintended tools or passing malicious inputs<br>- Keep a full, unalterable record of every tool call, what inputs were passed, and who approved it<br>- Ensure the connection between the agent and its tools is authenticated and secure | - Trade research assistant that retrieves live market data upon analyst approval<br>- KYC assistant that queries a customer record only after a compliance officer confirms the retrieval<br>- Legal assistant that pulls a contract clause from a document store on paralegal approval |
| **A2** | Workflow / Playbook Agent | The model can carry out a sequence of steps on its own, without needing human approval at each step. It operates within a defined set of allowed actions. If it encounters a situation outside that set, it stops and escalates to a human. | - Procedural autonomy: executes predefined procedures, no task decomposition<br>- Works from a fixed list of pre-approved actions it is allowed to take<br>- Can complete multi-step tasks without human approval at every step<br>- Must escalate to a human for anything outside its approved action list<br>- Limited memory within or across sessions<br>- Cannot access tools or systems outside its defined boundary<br>- Cannot hand off tasks to other agents | - Enforce the approved action list strictly; block and alert on any attempt to act outside it<br>- Ensure escalation paths cannot be bypassed or tampered with<br>- Give the agent access only to what it needs for its specific task<br>- Monitor the sequence of actions the agent takes to detect unexpected behaviour<br>- Verify the agent's connections to tools and services are authenticated<br>- Retain records of every checkpoint review as an auditable artefact | - Reconciliation agent that identifies breaks, executes standard corrections within a defined playbook, and escalates exceptions to operations<br>- Regulatory reporting agent that assembles filings within a defined template and submits only after human checkpoint sign-off<br>- Onboarding agent that executes a defined document collection sequence, escalating incomplete or ambiguous cases |
| **A3** | Goal-Directed Agent | The model is given a high-level goal and works out how to achieve it on its own. This includes breaking the goal into steps, deciding which tools or other agents to use, and coordinating their work. A human sets the goal and reviews progress at agreed milestones, but does not direct each step. | - Strategic autonomy: creates and chooses its own procedures to achieve a given goal<br>- Breaks down high-level goals into steps and sequences them independently<br>- Can direct and coordinate pre-approved, registered agents to carry out parts of the task<br>- Retains memory across sessions to track progress toward the goal<br>- Selects tools dynamically based on what the task requires, within approved limits<br>- Checks in with a human at defined milestones rather than at every step<br>- Can only use agents that have been registered and approved in advance<br>- Escalates to a human when confidence in a decision falls below a defined threshold | - Every communication between agents must be verified to confirm it comes from a trusted source<br>- Detect and block attempts to manipulate the agent through messages passed between agents or through tool outputs<br>- Prevent one agent from gaining access or permissions beyond what it has been granted, even when acting on behalf of another<br>- Protect stored memory from tampering or unauthorised access<br>- Verify the integrity of all tool connections across the entire network of agents, not just the lead agent<br>- Limit the scope of damage if any single agent is compromised or behaves unexpectedly<br>- Maintain a complete, unalterable record of every decision made, every task delegated, and every milestone reviewed<br>- All agents used must be registered, reviewed, and approved before they can be invoked | - Credit underwriting pipeline orchestrating document extraction, risk scoring, policy compliance checking, and exception flagging agents before presenting a recommendation for human approval<br>- Fraud investigation agent coordinating data retrieval, transaction pattern analysis, and case narrative drafting sub-agents<br>- Regulatory change management agent decomposing a new rule into impacted policy areas and delegating analysis to domain-specific sub-agents |
| **A4** | Self-Directing Agent | The model pursues objectives over an extended period with little to no human involvement during execution. It revises its own plan as circumstances change, dynamically extends its own tool set within defined policy bounds, and creates and retires agents as needed. The human sets the initial objective and is alerted only when something goes wrong. | - Revises its own goals and plan at runtime without waiting for human instruction<br>- Dynamically extends its own tool set within defined policy bounds when existing tools are insufficient<br>- Operates continuously over long periods without session boundaries<br>- Creates and retires agents dynamically to carry out and conclude parts of its work<br>- Maintains and updates its own memory over time<br>- Detects when something has gone wrong and corrects course without human prompting | - Verify the agent's behavioural boundaries before deployment and monitor them continuously during operation<br>- Detect unusual behaviour automatically and trigger controls to halt or limit the agent's actions<br>- Maintain tamper-proof, signed records of every plan change, tool onboarded, and agent created or retired<br>- Implement controls that can stop the agent immediately and that the agent itself cannot disable or circumvent<br>- Every decision the agent makes must be traceable and explainable to a human reviewer after the fact<br>- Design the system so that a failure or compromise of the agent cannot cascade across the broader organisation<br>- Maintain model risk management documentation consistent with SR 11-7 and equivalent regulatory frameworks | No FSI production system currently operates at full A4. Regulatory requirements around explainability, human oversight, and model risk management under SR 11-7 and the EU AI Act present barriers not yet resolved by the industry. A4 is defined here to provide taxonomic completeness and to inform governance frameworks being designed today for future deployment scenarios. |

## Summary

| Level | Name | Human Role | Action Scope | Memory | Multi-Agent | Regulatory Risk Tier |
|:---|:---|:---|:---|:---|:---|:---|
| **A0** | No Agency | Executes all actions | None | No memory | No | Minimal |
| **A1** | Tool-Calling Agent | Approves every action | Single-step, approved tools only | Session only | No | Low |
| **A2** | Autonomous Agent | Reviews at checkpoints | Fixed, pre-approved action list | Session and limited persistence | No | Moderate |
| **A3** | Goal-Directed Agent | Sets objective, reviews at milestones | Dynamic within approved limits | Persistent, cross-session | Yes, pre-approved registered agents only | High |
| **A4** | Self-Directing Agent | Sets objective, receives exception alerts | Self-determined within policy bounds | Persistent, self-managed | Yes, including dynamic creation and retirement of agents | Critical |

## Glossary

**Action**  
One tool invocation that produces one externally observable side effect. An action is atomic and non-composing. Downstream side effects that occur within the tool's own system are outside the scope of this classification. Where a business workflow is implemented behind a single tool call, that workflow is treated as a single action at the classification boundary.

**Procedural autonomy**  
The ability to execute a predefined sequence of steps without human approval at each step, but without the ability to create new procedures or decompose goals into new sub-tasks. Characteristic of A2.

**Strategic autonomy**  
The ability to independently decompose a goal into sub-tasks, choose between available procedures, and coordinate execution across tools and agents. Characteristic of A3 and above.

**State**  
Information the agent retains to track progress within or across tasks, such as case context, intermediate results, or task history. State persists to support continuity of work, not to modify the agent's own behaviour or decision rules.

**System state mutation**  
Any change made to an external system as a direct or indirect result of an agent's action. This includes writes to databases, API calls that trigger downstream processes, and any other externally observable change.
